## General Tips

A structured approach to solving problems (especially under interview pressure):

```
 ┌─────────────┐
 │ 1. GET INFO │  Constraints, input size, edge cases, expected output
 └──────┬──────┘
        ▼
 ┌──────────────┐
 │ 2. EXAMPLES  │  Work through examples. Is it a special/tricky case?
 └──────┬───────┘
        ▼
 ┌───────────────┐
 │ 3. BRUTE FORCE│  Get to brute-force ASAP. State runtime. Don't code yet.
 └──────┬────────┘
        ▼
 ┌──────────────┐   Look for unused info
 │ 4. OPTIMIZE  │   Solve it manually, then reverse engineer your thought process
 │   (BUDS)     │   Space vs time tradeoffs, hashing
 └──────┬───────┘   Bottlenecks · Unnecessary work · Duplication · Sorting
        ▼
 ┌──────────────┐
 │ 5. WALK      │  Walk through your approach in detail before coding
 └──────┬───────┘
        ▼
 ┌──────────────┐
 │ 6. CODE      │  Now implement it
 └──────┬───────┘
        ▼
 ┌──────────────┐
 │ 7. TEST      │  Start small → hit edge cases → stress test
 └──────────────┘
```

**Common patterns by input size:**

| Input size n | Expected complexity | Think about...              |
|-------------|--------------------|-----------------------------|
| n ≤ 20      | O(2^n), O(n!)      | Backtracking, bitmask       |
| n ≤ 500     | O(n^3)             | DP with 2D table            |
| n ≤ 10^4    | O(n^2)             | Nested loops, DP            |
| n ≤ 10^6    | O(n log n)         | Sorting, binary search      |
| n ≤ 10^8    | O(n)               | Two pointers, sliding window|
| n > 10^8    | O(log n), O(1)     | Math, binary search         |
